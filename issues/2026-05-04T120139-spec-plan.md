# ドメイン・業務プロセス記述によるGUI生成仕様（データモデリング統合版）

## 1. 結論

ドメインモデル（論理）を単一の入力（SSOT）とし、  
DBスキーマ・API・GUI・業務プロセスをすべて生成する。

---

## 2. 全体構造

```text
Model (SSOT)
  ├─ Entity
  │    ├─ Field
  │    ├─ Relation
  │    ├─ Constraint
  │    └─ Storage (hint)
  ├─ State
  ├─ Transition
  ├─ View
  └─ Rule

生成物:
  ├─ DB Schema
  ├─ Migration
  ├─ API
  ├─ Validation
  └─ GUI
```

---

## 3. エンティティ定義

```yaml
entities:
  ExpenseRequest:
    label: 経費申請

    fields:
      id:
        type: id
        primary: true

      amount:
        type: money
        required: true

      reason:
        type: text
        required: true

      applicant:
        type: ref
        target: User
        required: true

      status:
        type: state
        initial: draft

      createdAt:
        type: datetime
        system: true

      updatedAt:
        type: datetime
        system: true
```

---

## 4. フィールド仕様

```text
type:
  id / text / number / money / boolean
  date / datetime / enum / ref / state / file

属性:
  required: 必須
  primary: 主キー
  readonly: UI編集不可
  system: システム管理
  default: 初期値
```

## 8. 遷移

```yaml
transitions:
  submit:
    from: draft
    to: submitted

      - amount
      - reason
    guard: amount > 0

  approve:
    from: submitted
    to: approved
    role: manager

  reject:
    from: submitted
    to: rejected
    role: manager

      - rejectReason
```

## 受け入れ条件

- [ ] SSOT としての入力モデル方針が明文化されている
- [ ] 生成対象が DB / API / Validation / GUI / Workflow であることが明記
      されている
- [ ] 後続 issue が参照すべき基本用語と構成要素が揃っている
- [ ] この issue は実装タスク一覧ではなく基本計画であることが明記されて
      いる

## 非目標

- この issue 単体で全生成器を実装すること
- YAML 文法や runtime 挙動の細部をここですべて凍結すること
- 後続 issue の詳細 acceptance をこの issue 1 本に詰め込むこと

## 根拠

基本計画が曖昧なままだと、後続 issue ごとに SSOT や生成範囲の解釈がぶれる。
先に全体構造を固定しておくことで、個別 issue は局所仕様に集中できる。

## 12. DBスキーマ生成規則

```text
Entity → Table
Field → Column
ref → 外部キー
constraints → CHECK制約
state → ENUM or TEXT
system fields → 自動管理
```

例：

```sql
CREATE TABLE expense_requests (
  id TEXT PRIMARY KEY,
  amount INTEGER NOT NULL,
  reason TEXT NOT NULL,
  applicant_id TEXT NOT NULL,
  status TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (applicant_id) REFERENCES users(id),
  CHECK (amount > 0)
);
```

## 14. GUI生成規則

```text
Field → 入力部品
State → 画面状態
Transition → 操作ボタン
Rule → 活性条件
View → 編集/参照制御
```

---

## 15. バリデーション

```text
required → 必須チェック
constraint → 式評価
type → 型チェック
guard → 操作可否
```

---

## 16. 実行モデル

```text
1. Entity取得
2. 現在state判定
3. 利用可能transition列挙
4. rule評価
5. GUI生成
```

---

## 17. 非対象

```text
BPMN互換
並列ワークフロー
イベント駆動
タイマー処理
完全なレイアウト制御
```

## 18. 設計原則

```text
ドメインを先に定義する
UIは生成する
状態遷移を中心にする
制約は明示する
DBは結果として生成する
```
