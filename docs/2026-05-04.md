## 学んだこと

- **SSOT** は Single Source of Truth の略で、同じ情報を何度も別形式で持たず、一つの定義から複数の成果物を生成する考え方を指す
- 今回はいきなり API や GUI まで広げず、まず SQL という具体的な生成結果に絞ることで、schema IR の妥当性を検証しやすくした
- MoonBit では package ではなく directory が実際の compilation unit なので、同じ package の中なら file を分けても一つの実装として扱える
- **expression AST** は文字列をそのまま各 generator で再解釈するのではなく、一度意味のある木構造に変換してから共有する設計で、generator 間のズレを減らしやすい
- **API/action manifest** はまだ server code ではないが、entity と transition から route / action を機械可読な形で固定する中間成果物として有効
- **migration SQL** は schema 本体と別に up/down の境界を持つので、初期導入時の DB 反映手順を artifact として持ちやすい
- **GUI manifest** は実 UI コードそのものではないが、view ごとの field mode / action / enabled 条件を固定できるので renderer の入力として使いやすい
- **in-memory runtime** は HTTP server ではなく、まずは library として record と actor を受け取って判定する層を作る、という意味にした。これにより transport や storage を決める前に業務ルールを固められる
- generated manifest を再度 parse せず validated `Schema` / `Expr` を直接使うと、generator と runtime で意味のズレが起きにくい
