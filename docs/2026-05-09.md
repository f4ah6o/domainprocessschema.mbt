## audit event manifest

### 計画

- `docs/audit-event-manifest.md` を追加して、transition の成功・失敗 event shape を固定する
- failure event では structured diagnostics をそのまま `issues` に入れられるようにする
- README と schema contract から参照し、audit sink や履歴 UI との接続先を説明する

### やったこと

- `transition.applied` / `transition.rejected` の event kind を定義した
- `entity`、`recordId`、`transition`、`from`、`to`、`actor`、`input`、`result`、`issues`、`context` の共通 field を固定した
- rejected event の `issues` が parser / validator / runtime と同じ diagnostic shape を使うことを明記した
- audit sink、history UI、debug/replay への接続方針を書いた
- issue を `issues/closed/` へ移動し、resolution を残した

### 学んだこと

- audit event は runtime core の戻り値に直接入れるより、host が runtime outcome を正規化して sink へ渡す contract として定義する方が責務が分かりやすい
- event sourcing と audit log は似ているが、v1 では record rebuild を保証しない audit record として切る方が小さく閉じられる
