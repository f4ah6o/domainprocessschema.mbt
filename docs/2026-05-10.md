## structured diagnostics

### 計画

- validator / compiler / runtime が structured diagnostic shape を使っていることを確認する
- CLI / WASM demo / tests が `code` / `severity` / `target` / `message` / `hint` / `context` を前提にしていることを確認する
- roadmap の issue path を `issues/closed/` に合わせ、issue を close する

### やったこと

- `Diagnostic` と `DiagnosticSeverity` が public surface にあり、validator / parser / runtime helper が同じ shape を作ることを確認した
- tests が `UNKNOWN_STATE`、`INVALID_EXPR`、`INVALID_YAML_SYNTAX` などの fixed code と diagnostic field を検証していることを確認した
- WASM demo の editor/session JSON が diagnostic context と severity を JSON 化していることを確認した
- issue を `issues/closed/` に移動し、resolution を残した

### 学んだこと

- diagnostic は人間向け message だけでなく、agent や editor が安定判定するための `code` と `target` が重要になる
- `hint` と `context` を shape に含めておくと、同じ error code でも修正案や host metadata を後から安全に足せる
